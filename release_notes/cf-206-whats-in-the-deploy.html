Centralize service instance delete logic in ServiceInstanceDelete
02092df
- Splitting the locking logic between the service instance controller/ ServiceInstanceDeprovisioner and the delete action made it difficult to understand what was managing the locks for a given case, especially since the unlock would sometimes occur in a background job. - The delete action should be a single source of truth on how to delete a service instance. This will make it easier to implement features like recursive delete that do not go through the deprovisioner. - The behavior around when an 'async=true' job is locked has changed. The DeleterLock is meant to represent a broker operation in progress for that service instance. Previously the instance locked when the delete job was enqueued, however this resulted in more complicated logic that did not reflect the broker's actual state. Now the service instance is locked only when the job is executed - ServiceInstanceDelete needs the `ensure` since it might be wrapped in a timeout job that could kill it at any time. This prevents timed out operations from getting stuck 'in progress' - Move from service_instance.delete -> service_instance.destroy. There are no more destroy hooks, so it is safe to unify on a single deletion method. - Adjust NullClient deprovision to have a successful last operation. This was driven out by service purge and reduces the number of null checks in service instance delete. [#90815812]
